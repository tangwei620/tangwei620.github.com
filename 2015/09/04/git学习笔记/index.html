1. git概述

1.1 Git是目前世界上最先进的分布式版本控制系统。

2.1 Linus花了两周时间自己用C写了一个分布式版本控制系统，这就是Git。

2. 安装git

2.1 windows用msysgit安装

2.2 打开gitbash能看到美元符号就说明成果了

2.3 用下列命令设定用户名和密码。

$ git config --global user.name "Your Name"    
$ git config --global user.email "email@example.com"

3. 创建git库

3.1 选择一个合适路径，建立一个空文件夹，然后在空白处点击右键，选择git bash，输入git init，这样该路径下就会产生一个.git的文件夹，不要修改它。

3.2 千万不要使用Windows自带的记事本编辑任何文本文件。原因是Microsoft开发记事本的团队使用了一个非常弱智的行为来保存UTF-8编码的文件，他们自作聪明地在每个文件开头添加了0xefbbbf（十六进制）的字符，你会遇到很多不可思议的问题，比如，网页第一行可能会显示一个“?”，明明正确的程序一编译就报语法错误，等等，都是由记事本的弱智行为带来的。建议你下载Notepad++代替记事本，不但功能强大，而且免费！记得把Notepad++的默认编码设置为UTF-8 without BOM即可。

3.3 将文件放到Git仓库中，先用命令git add把文件添加到仓库，然后用git commit -m”comments”提交到仓库。

3.4 要掌握工作区的状态，使用git status。

3.5 要看文件具体哪里被修改了，使用git diff。

3.6 要看文件的历史，使用git log。如果想要精简显示，可以加上–pretty=oneline。

3.7 如果要回退到某个版本，使用git reset –hard HEAD^，上一个版本就是HEAD^，上上一个版本就是HEAD^^，当然往上100个版本写100个^比较容易数不过来，所以写成HEAD~100

3.8 另外，也可以使用commit_id过去未来穿梭到某个版本，详细命令是git reset –hard commit_id。可以使用git log查看历史，以确定回到哪个版本。要重返未来，使用git reflog查看命令历史，以便确定回来的commit_id。

4. 工作区和暂存区

4.1 电脑里直接能看到的目录，就叫工作区。

4.2 工作区下面的.git下面的index（或者叫stage）叫暂存区。

4.3 还有Git自动为我们创建的第一个分支master，以及指向master的一个指针叫HEAD。

4.4 因此，我们把文件添加到Git中意味着，先用命令git add把文件添加到暂存区，然后用git commit把暂存区的所有内容提交到当前分支。

4.5 每次文件修改后，如果不git add到暂存区，那就不能通过git commit加入到版本区中。

4.6 可以通过git diff HEAD – filename来查看工作区和版本区的区别。

5. 撤销修改

5.1当你改乱了工作区某个文件的内容，想直接丢弃工作区的修改时，用命令git checkout – file。

5.2 当你不但改乱了工作区某个文件的内容，还添加到了暂存区时，想丢弃修改，分两步，第一步用命令git reset HEAD file，就回到了场景1，第二步按5.1操作。

5.3 已经提交了不合适的修改到版本库时，想要撤销本次提交，参考版本回退一节，不过前提是没有推送到远程库。

6. 删除文件

6.1 首先在文件管理器中把没用的文件删了，或者用rm命令，git status命令会立刻告诉你哪些文件被删除了，现在你有两个选择，一是确实要从版本库中删除该文件，那就用命令git rm删掉，并且git commit。另一种情况是删错了，使用命令git checkout – filename来恢复，只能恢复文件到最新版本，你会丢失最近一次提交后你修改的内容。

7. 远程仓库基础知识

7.1 注册GitHub账号，就可以免费获取远程仓库。当然，GitHub是国外的，中国有一个Git@OSC，都是提供远程仓库服务的。

7.2 有了GitHub账号之后，需要创建SSH Key，在用户主目录下，看看有没有.ssh目录，如果有，再看看这个目录下有没有id_rsa和id_rsa.pub这两个文件，如果已经有了，可直接跳到下一步。如果没有，打开Shell（Windows下打开Git Bash），创建SSH Key：ssh-keygen -t rsa -C “youremail@example.com”
。

7.3 .ssh目录，里面有id_rsa和id_rsa.pub两个文件，这两个就是SSH Key的秘钥对，id_rsa是私钥，不能泄露出去，id_rsa.pub是公钥，可以放心地告诉任何人。

7.4 登陆GitHub，打开“Account settings”，“SSH Keys”页面：点“Add SSH Key”，填上任意Title，在Key文本框里粘贴id_rsa.pub文件的内容：

7.5 如果有多台电脑需要提交，允许将每台电脑的Key添加到GitHub中。

8. 添加远程仓库

8.1 要关联一个远程库，使用命令git remote add origin git@server-name:path/repo-name.git。

8.2 关联之后，第一次推送使用命令git push -u origin master，推送master分支所有内容。

8.3 此后，每次修改就可以使用命令git push origin master，就可以推送最新修改。

8.4 如果要克隆一个远程仓库到本地，必须要知道仓库的地址，使用git clone命令进行克隆。

8.5 仓库的地址有两种写法，一种是用https的，一种是用ssh的，ssh更快，但有些公司无法使用ssh。

9. 分支管理

9.1 一开始的时候，master分支是一条线，Git用master指向最新的提交，再用HEAD指向master，就能确定当前分支，以及当前分支的提交点：

9.2 当我们创建新的分支，例如dev时，Git新建了一个指针叫dev，指向master相同的提交，再把HEAD指向dev，就表示当前分支在dev上：

9.3 从现在开始，对工作区的修改和提交就是针对dev分支了，比如新提交一次后，dev指针往前移动一步，而master指针不变：

9.4 Git的合并就是直接把master指向dev的当前提交，就完成了合并：

9.5 合并之后，就可以删除dev分支

9.6 以上用到命令有：

查看分支：git branch
创建分支：git branch <name>
切换分支：git checkout <name>
创建+切换分支：git checkout -b <name>
合并某分支到当前分支：git merge <name>
删除分支：git branch -d <name>

10. 解决冲突

10.1 如果两个分支修改了同一个文件，修改的内容彼此冲突，当我们再次使用git merge <name>时，就会提示文件存在冲突，必须手动解决冲突才能提交，用git status可以告诉我们冲突的文件。

10.2 可以用git log --graph --pretty=oneline --abbrev-commit查看分支的合并情况。

11.3 所以在团队开发时，应采用不使用fast forward的合并模式，从而保留分支信息。

11.4 在实际开发中，我们应该按照几个基本原则进行分支管理：首先，master分支应该是非常稳定的，也就是仅用来发布新版本，平时不能在上面干活；那在哪干活呢？干活都在dev分支上，也就是说，dev分支是不稳定的，到某个时候，比如1.0版本发布时，再把dev分支合并到master上，在master分支发布1.0版本；你和你的小伙伴们每个人都在dev分支上干活，每个人都有自己的分支，时不时地往dev分支上合并就可以了。所以，团队合作的分支看起来就像这样：

12.Bug分支

12.1 一般情况下，修复Bug，就在master或dev上新建一个bug分支，然后合并，然后删除bug分支。

12.2 如果手头的工作没有做完，可以先把工作现场git stash一下，然后按照一般情况去修复bug，修复后，再git stash pop，回到工作现场。

12.3 可以多次stash，恢复的时候，先用git stash list查看，然后恢复指定的stash，用命令：git stash apply stash@{0}。

12.4 有两种恢复方法：一是用git stash apply恢复，但是恢复后，stash内容并不删除，你需要用git stash drop来删除；另一种方式是用git stash pop，恢复的同时把stash内容也删了：

13.Feature分支

13.1 增加一个新功能跟修改bug是一样的
13.1 但如果要丢弃一个没有被合并过的分支，可以通过git branch -D <name>强行删除。

14.多人协作

14.1 查看远程库信息，使用git remote -v，如下图所示

$ git remote -v
origin  git@github.com:michaelliao/learngit.git (fetch)
origin  git@github.com:michaelliao/learngit.git (push)

如果没有推送权限，就看不到push的地址。

14.2 本地新建的分支如果不推送到远程，对其他人就是不可见的；用git push origin master推送主分支，用git push origin branch-name推送其他分支。

14.3 分支需要推送的原则如下：

master分支是主分支，因此要时刻与远程同步；
dev分支是开发分支，团队所有成员都需要在上面工作，所以也需要与远程同步；
bug分支只用于在本地修复bug，就没必要推到远程了，除非老板要看看你每周到底修复了几个bug；
feature分支是否推到远程，取决于你是否和你的小伙伴合作在上面开发。

14.4 从本地推送分支，使用git push origin branch-name，如果推送失败，先用git pull抓取远程的新提交；

14.5 在本地创建和远程分支对应的分支，使用git checkout -b branch-name origin/branch-name，本地和远程分支的名称最好一致；

14.6 如果git pull提示“no tracking information”，则说明本地分支和远程分支的链接关系没有创建，用命令git branch --set-upstream branch-name origin/branch-name，本地和远程分支的名称最好一致；

14.7 从远程抓取分支，使用git pull，如果有冲突，要先处理冲突。

15.标签管理

15.1 命令git tag <name> [`commit id]用于新建一个标签，默认为HEAD，也可以指定一个commit id，可以用git log --pretty=oneline --abbrev-commit查看commit id；

15.2 命令git tag -a <tagname> -m "blablabla..."用于创建一个带有信息的标签；

15.3 可以用git tag和git show <tagname>查看标签；

15.4 命令git tag -d <tagname>用于删除标签；

15.5 命令git push origin <tagname>推送某个标签到远程，或者用命令git push origin --tags一次性推送全部尚未推送到远程的本地标签；

15.6 如果标签已经推送到远程，要删除远程标签就麻烦一点，先从本地删除git tag -d <tagname>，然后，从远程删除。删除命令也是push，但是格式如下：git push origin :refs/tags/<tagname>

16.GitHub使用

16.1 如果从其他作者的仓库地址，如git@github.com:twbs/bootstrap.git克隆，因为没有权限，你将不能推送修改。

16.2 如果你想参与别人的项目，点“Fork”就在自己的账号下克隆了一个别人的项目仓库，然后，从自己的账号下clone到本地，如果你想修复别人项目的bug，先往自己的GitHub上推送，如果你希望别人接受你的修改，你可以在GitHub上发起一个pull request。当然，对方是否接受你的pull request就不一定了。示意图如下：

17.自定义Git

17.1 忽略某些文件时，需要编写.gitignore；

17.2 .gitignore文件本身要放到版本库里，并且可以对.gitignore做版本管理！

17.3 配置别名，就可以输入命令时偷个懒。我们只需要敲一行命令，告诉Git，以后st就表示status：git config --global alias.st status

18.搭建Git服务器

18.1 搭建Git服务器需要准备一台运行Linux的机器，强烈推荐用Ubuntu或Debian，这样，通过几条简单的apt命令就可以完成安装。假设你已经有sudo权限的用户账号，具体安装细节请自行百度。
